# 摆摊了么 - 测试策略方案

> 适用于单人开发的高效测试方案，减少重复调试时间

---

## 一、现状分析

### 技术架构
- **前端**：微信小程序原生框架（WXML + WXSS + JS）
- **后端**：微信云开发（CloudBase）
- **数据库**：云数据库（MongoDB）
- **存储**：云存储

### 测试难点
1. 云函数依赖云环境，本地难以完全模拟
2. 地图组件、图片上传等需要真机测试
3. 用户权限（普通用户 vs 管理员）需要多账号切换
4. 一人开发，没有专职测试人员

---

## 二、分层测试策略

### 第1层：云函数单元测试（投入产出比最高）

**为什么先做这层？**
- 云函数是纯 JavaScript，最容易测试
- 业务逻辑集中在这里，出bug影响最大
- 可以在本地用 Node.js 快速测试

**测试范围**
```
cloudfunctions/
├── getStalls/           # 查询逻辑（筛选、排序、分页）
├── getStallDetail/      # 详情查询 + 访问计数
├── submitApplication/   # 数据校验、图片处理
├── adminGetApplications/# 数据聚合（关联用户表）
├── auditApplication/    # 状态流转
├── submitFeedback/      # 反馈提交
└── ...
```

**测试重点**
| 云函数 | 关键测试点 |
|--------|-----------|
| getStalls | 分类筛选、关键词搜索、分页逻辑、坐标距离计算 |
| submitApplication | 字段校验、图片压缩、重复提交拦截 |
| adminGetApplications | 图片URL转换、用户数据关联 |
| auditApplication | 状态机流转（待审核→通过/拒绝） |

**推荐工具**
- **Jest**：最主流的 JS 测试框架
- **miniprogram-automator**：微信官方小程序自动化测试（可用于云函数集成测试）

**测试数据管理**
- 使用独立的测试数据库（云开发支持多环境）
- 编写数据初始化脚本，每次测试前重置
- 关键：不要在生产环境跑测试！

---

### 第2层：API 接口契约测试

**目的**：确保前端调用和后端返回格式一致

**测试方法**
1. 定义每个云函数的输入输出格式（JSON Schema）
2. 编写接口文档（可以用注释直接生成）
3. 测试用例覆盖正常情况和异常情况

**示例测试场景**
```
submitApplication
├── 正常情况：完整数据提交 → 返回成功
├── 边界情况：最少必填字段 → 返回成功
├── 异常情况：缺少必填字段 → 返回具体错误信息
├── 异常情况：图片超过3张 → 返回校验错误
└── 异常情况：图片上传失败 → 优雅降级
```

---

### 第3层：前端关键路径自动化（适度投入）

**适合自动化的场景**
- 入驻申请表单填写流程
- 管理员审核操作流程
- 分页加载更多逻辑

**不适合自动化的场景**（建议手动测试）
- 地图交互（缩放、标记点点击）
- 图片上传（涉及相机/相册权限）
- 地理位置获取
- 扫码功能

**推荐工具**
- **miniprogram-automator**：微信官方出品，支持模拟器/真机
- **jest + jest-environment-node**：配合云函数测试

---

### 第4层：关键用户场景端到端测试

**核心场景清单**（优先级排序）

| 优先级 | 场景 | 测试目的 |
|--------|------|----------|
| P0 | 用户查看附近摊位 → 点击查看详情 → 导航 | 核心使用流程 |
| P0 | 摊主填写入驻申请 → 上传图片 → 提交 | 核心业务流程 |
| P0 | 管理员审核通过 → 摊位显示在地图 | 核心业务流程 |
| P1 | 用户反馈摊位信息有误 → 管理员处理 | 风险控制流程 |
| P1 | 摊主扫码确认摊位 → 可信度更新 | 数据维护流程 |
| P2 | 筛选功能（分类 + 时间） | 辅助功能 |

**测试方式**
- 开发阶段：每次修改后手动走一遍 P0 场景
- 发版前：完整走一遍所有场景
- 可以考虑录制操作脚本，半自动化回归

## 三、单人开发的高效测试技巧

### 1. 本地调试优先

**云函数本地调试**
- 微信开发者工具支持云函数本地调试
- 使用 `wx-server-sdk` 的 mock 模式
- 关键：准备好模拟数据，不要每次都走完整流程

**Mock 数据策略**
```
mock/
├── stalls.js          # 模拟摊位数据（10-20条）
├── applications.js    # 模拟申请数据
├── users.js           # 模拟用户信息
└── index.js           # 统一导出
```

### 2. 测试账号管理

**建议准备3个测试账号**
- 账号A：普通用户（体验游客视角）
- 账号B：待审核摊主（体验入驻流程）
- 账号C：管理员（体验后台功能）

**快速切换技巧**
- 使用微信开发者工具的「多账号调试」功能
- 真机测试时准备2-3部手机，或用小号

### 3. 关键检查点清单

**每次改完代码检查这些**

云函数修改后：
- [ ] 部署到云端了吗？
- [ ] 数据库权限配置正确吗？
- [ ] 新字段有默认值吗？
- [ ] 错误处理有日志记录吗？

前端修改后：
- [ ] 真机测试了吗？（模拟器不等于真机）
- [ ] 低版本基础库兼容吗？
- [ ] 网络异常情况处理了吗？
- [ ] 加载状态/空状态处理了吗？

### 4. 回归测试策略

**核心原则**：不要每次都全量测试

**建立测试套件**
```
smoke-test/              # 冒烟测试（5分钟）
├── 首页地图加载正常
├── 列表筛选正常
├── 详情页显示正常
└── 申请页提交正常

regression-test/         # 回归测试（30分钟）
├── 所有P0场景
└── 最近修改的相关功能

full-test/               # 全量测试（2小时）
├── 所有功能点
└── 异常流程
```

## 四、自动化测试实施建议

### 阶段一：基础建设（1-2天）

1. **搭建 Jest 测试环境**
   - 安装 jest 及相关依赖
   - 配置测试数据库连接
   - 编写测试数据初始化脚本

2. **挑选2-3个核心云函数写测试**
   - 优先选：getStalls、submitApplication
   - 覆盖正常流程 + 1-2个异常情况

### 阶段二：逐步完善（持续迭代）

1. **为每个新功能补充测试**
   - 先写测试，再写代码（TDD 思路）
   - 至少覆盖核心逻辑

2. **建立 CI 流程（可选）**
   - 提交代码前自动跑测试
   - 部署云函数前自动跑测试

### 阶段三：效率优化（视情况）

1. **前端自动化测试**
   - 只覆盖最稳定、最核心的流程
   - 不要追求100%覆盖率，ROI太低

## 五、测试用例设计示例

### 示例1：submitApplication 云函数测试用例

```
测试套件：提交入驻申请

用例1.1：完整数据提交成功
- 输入：所有必填字段 + 2张图片 + 联系方式
- 期望：返回成功，数据库新增记录，图片存储到云存储

用例1.2：最小数据提交成功
- 输入：仅必填字段，无图片，无联系方式
- 期望：返回成功，数据库新增记录

用例1.3：缺少必填字段失败
- 输入：缺少 categoryId
- 期望：返回错误码，提示具体缺失字段

用例1.4：图片超过限制失败
- 输入：上传4张图片
- 期望：返回校验错误

用例1.5：重复提交拦截
- 输入：同一用户在24小时内第二次提交
- 期望：返回提示，建议修改已有申请
```

### 示例2：getStalls 云函数测试用例

```
测试套件：获取摊位列表

用例2.1：无条件查询
- 输入：空参数
- 期望：返回所有已上架摊位，按可信度排序

用例2.2：分类筛选
- 输入：categoryId = "烧烤"
- 期望：只返回烧烤类摊位

用例2.3：关键词搜索
- 输入：keyword = "老王"
- 期望：返回摊位名或描述包含"老王"的摊位

用例2.4：分页加载
- 输入：page = 2, pageSize = 10
- 期望：返回第11-20条数据

用例2.5：组合条件
- 输入：categoryId + keyword + 分页
- 期望：正确过滤并分页返回
```

## 七、时间投入建议

作为单人开发者，建议测试时间占比：**20-30%**

**每周时间分配示例（假设每周20小时开发）**
- 功能开发：14小时
- 编写/运行单元测试：3小时
- 手动集成测试：2小时
- Bug修复 + 回归：1小时

**关键投入点（按优先级）**
1. 核心业务逻辑单元测试（getStalls、submitApplication）
2. 建立 Mock 数据集（减少重复造数据时间）
3. 关键用户场景检查清单
4. 其他功能按需补充测试

---

## 八、总结

**核心原则**
1. **优先测试云函数**：业务逻辑在这里，纯JS好测试
2. **Mock 数据先行**：准备好测试数据，事半功倍
3. **关注核心场景**：不要追求100%覆盖，抓重点
4. **真机测试不可少**：模拟器无法完全替代真机
5. **建立检查清单**：避免重复劳动和遗漏

**快速启动建议**
1. 先为 `getStalls` 和 `submitApplication` 写5-10个测试用例
2. 准备一套 Mock 数据（10个摊位、5个申请）
3. 建立 P0 场景检查清单，每次改代码跑一遍
4. 逐步完善，不要追求一步到位
